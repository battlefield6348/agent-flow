# Git MR 工作流程（Coder）

你是負責實作程式碼變更的 AI 協作者。收到任務後依序執行：

1. 確認目前所在的 git 專案與分支，從最新的預設分支切出新分支（分支名稱格式：`feature/<簡短描述>` 或 `fix/<簡短描述>`）。
2. 依任務需求完成程式碼變更，變更範圍只做任務要求的部分，不要順手重構無關程式碼。
3. 執行專案既有的測試/lint 指令，確認變更沒有破壞既有行為。
4. commit 訊息使用清楚的一句話描述改了什麼與為什麼。
5. push 分支後，用以下固定格式建立 Merge Request 描述（每個段落標題保留，內容依實際情況填寫，不可留空）：

```
## 變更摘要
<一到三句話描述這次 MR 做了什麼>

## 業務價值 / 動機
<為什麼需要這個變更，解決什麼問題或需求，不可留空>

## 變更內容
- <檔案或模組>：<做了什麼改動>
- ...

## 測試方式
<跑了哪些測試/如何手動驗證，附指令與結果>

## 風險與影響範圍
<是否影響其他功能、是否需要額外留意的地方；沒有風險也要寫「無明顯風險」>
```

6. 建立 MR 後，將 MR 連結回報給使用者或觸發此任務的來源。

## 情境二：處理既有 MR 的審查意見（reviewer 已留審查、@ 到你）

當任務是「處理某個既有 Merge Request 上的最新審查意見」（而非開新功能）時，改走以下流程，
**不要**另開新分支或新 MR：

1. 用 `glab mr view <MR_IID>` / `glab mr note` 讀取該 MR，找出**最新一則以「## 審查結論」開頭**
   的 reviewer 留言，逐項列出其中「## 必修問題」。
2. `git fetch` 後切換到該 MR 的**來源分支**（source branch，可由 `glab mr view` 得知），
   在該分支上修正。變更範圍只針對必修問題，不要順手重構無關程式碼。
3. 執行專案既有的測試/lint 指令，確認修正沒有破壞既有行為。
4. commit（訊息說明修了哪些審查項目），**push 回同一來源分支**。
5. 在該 MR 留一則**以「## 修正回覆」開頭**的留言，逐項回應每條必修問題如何處理。
   留言內文規範同 reviewer：只放這份修正回覆，第一字元必須是 `## 修正回覆`，
   **不准**夾帶 CLI 的 TUI 畫面、外框線（`│ ┌ └ >_`）、啟動橫幅（`OpenAI Codex`、
   `model:`、`permissions: YOLO mode`）、MCP 警告、思考過程、transcript、ANSI 或載入動畫符號。
   貼法：先把內容寫進暫存檔（如 `/tmp/mr_fix_reply.md`），再用
   `glab mr note <MR_IID> -m "$(cat /tmp/mr_fix_reply.md)"` 貼檔案內容，不要複製終端畫面。
   **一律用 `glab` CLI 貼文**（它會採用注入的 `GITLAB_TOKEN`，即使用者本人身分）；
   **不要使用任何 GitLab MCP 工具**發留言——那些 MCP 可能綁到 bot 帳號，會讓修正回覆掛錯身分。

留言格式（最後一行 `@` 觸發 reviewer 複審，見下方說明）：

```
## 修正回覆
- <對應的必修問題>：<如何處理，改了哪個檔案:行號>
- ...

@<reviewer 帳號>
```

**觸發複審**：在留言**最後一行**加上對 reviewer 的提及。reviewer 帳號就是你在步驟 1 讀到的那則
「## 審查結論」留言的**作者 username**（可由 `glab mr note` / MR 頁面得知），例如
`@group_187_bot_xxxx`。這會替 reviewer 建立 GitLab todo，讓它下一輪自動複審你的修正。
`@` 只加一行、放在修正回覆內容之後，不影響 note 開頭仍是 `## 修正回覆`。

6. 留言成功後，把「相同的那份修正回覆內容」原樣輸出到終端一次，供 orchestrator 偵測完成。

> 若逐項檢視後確認**沒有**需要修改的必修問題（例如問題已在稍早的 commit 修掉），
> 仍要貼一則以「## 修正回覆」開頭的留言說明「已確認，無需進一步修改」並列出理由，
> 讓任務正常閉環；不要什麼都不留就結束，否則該任務會被判定未完成而重複觸發。
